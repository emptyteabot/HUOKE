## 一句话结论

LeadPulse 是一个面向小型服务团队、留学咨询、代运营工作室、独立开发者和早期出海团队的轻量获客产品。它试图解决的不是“再做一个 CRM”或“再做一个泛 AI 工具”，而是一个更靠近现金流的问题：从公开平台、评论区、帖子、社群讨论、搜索结果和已有网页里，找到正在表达需求的人，把这些人整理成可筛选、可导出、可触达、可成交推进的线索，然后用软件版或代跑版变现。

更直白地讲，LeadPulse 的核心承诺是：不要先买一堆复杂工具，也不要靠泛流量碰运气，先看一小批真实样本，确认这些人是不是真的在问“找谁做、多少钱、哪个好、怎么避坑、有没有推荐”。如果样本方向对，客户可以自己用软件筛名单、导名单、联系，也可以让你代跑一轮，交付一版整理好的结果和首轮沟通建议。

当前产品不是一个单一形态。仓库里同时存在早期 Streamlit 留学获客系统、Node/Express 后端、Electron 打包配置、爬虫脚本、Gemini/DeepSeek 线索生成脚本、OpenClaw 数据采集脚本、以及目前最接近线上站点的 Next.js 前台 `frontend-b2b`。真正对应 `leadpulse.cc.cd` 的主要是 `frontend-b2b`。线上站点和 GitHub 当前 `main` 属于同一条产品线、同一套应用结构，但线上部署不是当前仓库 HEAD 的精确版本，线上标题、部分文案和数据源仍停留在较早一版。

## 当前产品处在什么阶段

LeadPulse 不是“只有 PPT 的想法”，也不是“已经完全稳定的成熟 SaaS”。它处在一个很典型的早期商业验证阶段：有能打开的官网，有公开产品页和定价页，有可访问的线索 API，有内部仪表盘，有导出、积分、开通码、付款意向、消息草稿、任务跟进、交付包等模块，有真实或样例化的数据文件，也有不少历史遗留代码和命名漂移。

这个阶段最重要的判断不是“代码是不是整洁”，而是它有没有一个可卖的最小闭环。答案是有，但要谨慎使用。它已经能展示价值、收集意向、展示线索、隐藏链接、导出 CSV、生成触达草稿、登记付款意向、处理启动码和内部任务。但它离“安全稳定公开规模化卖”还有距离，主要差在权限边界、部署一致性、数据来源合规、后端依赖健康、密钥治理、真实支付闭环和交付 SOP 固化。

从商业角度看，它更适合作为“服务带软件”的早期产品，而不是一上来就定位成纯自助 SaaS。原因很简单：数据质量、平台合规、线索筛选和成交推进都高度依赖人工判断。客户真正愿意付钱的往往不是“我能看到一个表格”，而是“你能不能帮我找到一批更值得联系的人，并告诉我怎么联系”。所以当前最顺的商业路径是免费样本拉信任、¥99 软件版做低门槛验证、¥999 代跑版承接结果型需求。

## 目标用户

LeadPulse 当前最适合以下几类人：

第一类是留学咨询、教育咨询、文书工作室、申请服务小团队。这类业务天然依赖高意向需求，用户会在小红书、知乎、微博、豆瓣、论坛等地方问“留学中介怎么选”“有没有靠谱文书老师”“申请预算多少”“某某机构靠谱吗”。这些提问天然就是意向信号，远比冷启动广告流量更接近成交。

第二类是小型服务团队和代运营工作室。包括小红书代运营、短视频代运营、B2B 内容服务、网站搭建、AI 自动化咨询、营销顾问、财税法务咨询等。它们的问题不是没有产品，而是缺稳定的一批潜在客户。公开平台上已经有人在问方案、问报价、问推荐，只是这些问题分散在评论区、帖子和搜索结果里，人工长期盯很累。

第三类是独立开发者、早期 SaaS 团队、indie hacker、出海小团队。这类用户不一定需要留学线索，但需要“从公开讨论里找需求”的能力。例如有人在 Reddit、X、Product Hunt、微信群、小红书、知乎上抱怨某类工具不好用，或者问有没有某种替代方案。LeadPulse 的方法论可以迁移到这些场景，只是关键词、平台和触达方式要换。

第四类是还没有成熟销售团队的小老板。他们未必会使用复杂 CRM，也不想先配置 Salesforce、HubSpot、Apollo、Clay、Zapier、SendGrid、Stripe 等一整套工具。他们更关心“今天能不能拿到 20 条看起来能联系的人”。LeadPulse 当前文案里的“先看样本，再决定自己做还是交给我们代跑”，正是对这类用户更自然的表达。

不太适合的用户也要说清楚。它不适合已经有成熟销售组织、严格采购流程、复杂权限体系的大公司；不适合只想做品牌舆情监控的人；不适合期待完全自动化替代销售的人；不适合希望大量无边界抓取平台数据再群发骚扰的人。LeadPulse 的正确边界应该是“辅助发现公开意向信号和整理首轮触达建议”，不是“绕过平台规则做批量骚扰”。

## 产品真正卖的是什么

LeadPulse 表面上卖的是线索筛选工具，底层卖的是获客不确定性的降低。客户付钱之前最怕三件事：第一，线索是不是假的；第二，拿到线索以后能不能联系；第三，我为什么要信你能帮我找到客户。

因此产品应该围绕三层价值组织。

第一层是“样本信任”。免费样本不是福利，而是销售武器。早期客户很难因为一个官网就信任你，但他能通过 5 条真实样本判断方向对不对。样本里必须能看到平台、作者、关键词、意图分数、需求原文、是否可私信、为什么判定为目标客户。样本不一定要多，但必须真实、有上下文、能解释。

第二层是“名单效率”。如果客户自己跑，他要的是一批经过筛选的名单，而不是原始搜索结果。LeadPulse 应该帮助他筛掉同行、机构号、广告号、营销号、噪声样本，优先保留真正提问、表达预算、急迫时间线、明确求推荐、可私信或可继续追踪的人。这个价值可以用软件版承接。

第三层是“成交推进”。很多客户拿到名单也不会联系，或者会用很差的话术群发。LeadPulse 的 AI 草稿、消息库、任务跟进、代跑服务，价值就在这里。它要帮客户把“看到一个帖子”变成“发出一条合理的首轮私信/邮件”，再变成“预约/咨询/成交机会”。这个价值可以用代跑版承接。

## 当前公开站点体验

线上站点 `https://leadpulse.cc.cd` 当前能打开，服务由 Cloudflare 前置。它是一个 Next.js 站点，HTML 中有 `_next/static` 资源，API 路由也由同一个 Next 应用提供。

公开首页强调的是从公开平台里找到正在找方案的人。首页不是传统营销官网的大而全介绍，而是把产品压缩成几个判断点：适合谁、怎么开始、为什么先试一轮、为什么不是又一个看板工具。这个方向是对的，因为产品早期不应该先讲复杂技术，而应该讲“你能不能先拿到一批值得联系的人”。

`/product` 是产品介绍页。它解释“我们帮你做什么”：从公开平台找出已经在问价格、推荐、怎么选的人；筛掉明显同行、机构号和噪声；整理成能继续联系的名单。它也说明适合哪些人和不适合哪些情况。这个页面的语气比较克制，没有过度许诺“自动成交”。

`/pricing` 是收费方式页。当前方案是免费样本、软件版和代跑版。仓库当前 HEAD 的定价是：免费样本 ¥0/次，软件版 ¥99/月，代跑版 ¥999/首轮。这个定价很早期，也很适合验证需求。¥99 的作用是降低第一次付款门槛，不一定承担最终商业模型；¥999 的代跑版更接近实际利润来源。

`/faq` 回答常见问题，适合降低购买前疑虑。`/book` 承接 15 分钟沟通。`/register?plan=free` 会重定向到 `/start?plan=free`。`/pay?plan=pro` 展示付款确认路径，当前主要是微信收款码和启动码，不是默认全自动 Stripe。

`/dashboard`、`/dashboard/leads`、`/dashboard/messages`、`/dashboard/tasks` 和 `/ops` 已经被 internal access 门禁保护。未带正确 access key 时返回 Internal Access 页面。这说明线上已经不是完全裸奔的后台。但 API 的权限和页面权限不是一回事，之前我发现 `/api/export` 可以被直接调用导出 CSV，这已经在本地修复提交中处理。

## 当前仓库结构

仓库根目录包含多个历史阶段的产物。理解它时不要把它当成一个单纯 Next 项目。

`frontend-b2b` 是当前最重要的目录。它是 Next.js 15 + React 19 应用，包含公开官网、定价、支付确认、内部后台、线索 API、导出 API、兑换码、付款意向、消息草稿、任务、智能评估、社区情报、交付包等模块。它对应 `leadpulse.cc.cd` 的线上产品。

`backend` 是 Node/Express + Prisma 后端，包含 auth、leads、emails、campaigns、ai、stripe 等路由。它看起来是早期想做标准 SaaS 后端时留下的部分。目前后端构建存在 Prisma 7 和旧 schema 写法不兼容、Stripe API 版本类型不匹配、LinkedIn 服务 DOM 类型缺失等问题。因此它不是当前线上主路径。

`streamlit-app` 是早期纯 Python/Streamlit 版本，聚焦留学机构场景，包含用户登录、学生管理、AI 生成邮件、邮件发送、邮件追踪、工作流、分析、平台爬虫等功能。这部分更像第一代 MVP 或内部工具，未必是当前公开销售页主推版本。

`scrapers`、`openclaw_*`、`xiaohongshu_*`、`china_social_scraper.py`、`scraper_manager.py` 等文件是数据采集相关代码。它们代表 LeadPulse 的“供给侧”：怎么从公开平台拿到原始候选内容。

`tools` 目录包含导出、冷启动包、手动工作台、自用增长循环、OpenClaw 调度等脚本。它们是运营和自用工具，适合 founder 自己跑，不一定要暴露给客户。

`data` 和 `frontend-b2b/data` 里有线索快照、自增长报告、账号、外联队列等数据。线上 `/api/leads` 当前使用的是 `frontend-b2b/data/leads_snapshot.json` 一类 bundled snapshot 数据源。这个数据源能让网站在没有 Supabase、OpenClaw 最新文件或本地 exporter 的情况下仍能展示样本。

根目录大量 Markdown 文档记录了不同阶段的定位、部署、交付、竞品、留学冷启动和 Gemini workflow。这些文档有价值，但存在历史口径混杂：有些叫 GuestSeek，有些叫 LeadPulse，有些强调留学，有些强调 B2B SaaS，有些讲 Streamlit，有些讲 Next 前台。以后需要整理成一份主文档，其他旧文档归档。

## `frontend-b2b` 的核心架构

`frontend-b2b` 是 App Router 结构。公开页面在 `app` 下，组件在 `components` 下，业务逻辑在 `lib` 下，API route 在 `app/api` 下。

`app/page.tsx` 是首页。它使用 `SiteHeader` 和 `SiteFooter`，核心文案是“帮你从公开平台里找到正在找方案的人”。它列出适合谁、怎么开始、价值点和为什么先试一轮。

`app/product/page.tsx` 是产品介绍页。`app/pricing/page.tsx` 是定价页，定价来自 `lib/pricing.ts`。`app/pay/page.tsx` 是付款确认页，展示微信收款码、方案亮点、兑换码输入和购买前常见问题。`app/register/page.tsx` 只是重定向到 `/start`。

`middleware.ts` 控制内部页面访问。`/dashboard/:path*`、`/ops`、`/platform`、`/demo`、`/security`、`/agents`、`/proof`、`/investors`、`/cases`、`/experiments` 等路径需要 internal access key 或被软隐藏。公开生产环境必须配置 `LEADPULSE_INTERNAL_ACCESS_KEY`，否则非本地访问会返回 503 或重写到 internal login。

`app/api/health/route.ts` 返回 `{status:"ok", ts:...}`。这是部署健康检查。

`app/api/leads/route.ts` 是线索 API 的核心。它加载线索、打分、过滤、隐藏链接、返回 entitlements 和 wallet token。数据源优先级大致是：本地 Python exporter、OpenClaw 最新 JSON、Supabase、bundled snapshot。线上当前返回的 `source` 是 `bundled_snapshot`，说明生产环境没有接上最新本地 exporter 或 OpenClaw 文件，也没有走 Supabase。

`app/api/export/route.ts` 是导出 CSV。导出时会根据 wallet 的免费次数或 credits 扣减并解锁链接。我已经在本地提交中加了导出前访问检查：没有免费次数、没有积分就返回 402，不再先构建导出数据，也不会返回 CSV。

`lib/lead_wallet.ts` 是轻量钱包/额度系统。它用 HMAC 签名 cookie/token，记录 user_id、credits、exports_count、free_exports_used、last_export_at、links_unlocked_until。这个设计很轻，不依赖数据库，适合早期验证。但它本质上是客户端携带状态，必须有强 `WALLET_SIGNING_SECRET`。我已经改成生产环境必须配置该密钥，默认免费导出为 0。

`app/api/payment-intent/route.ts` 处理付款意向。它会创建 intake record、智能评估、任务、消息草稿、通知。如果 `LEADPULSE_PAYMENT_PROVIDER` 不是 `stripe`，默认走微信/人工核验路径；如果设置为 `stripe`，会创建 Stripe Checkout Session。当前产品公开文案也承认“不把人工确认包装成完全自动支付”。

`app/api/redeem-code/route.ts` 处理兑换码，底层调用 `lib/redemption.ts`。兑换成功后返回 startUrl、deliveryId、accessCode、workspaceId、planName。兑换码适合当前“微信付款后人工发码”的商业流程。

`lib/storage.ts` 用 Node SQLite 的 `DatabaseSync` 做本地状态存储，数据文件在 `../data/leadpulse_state.sqlite`。这是早期本机/单机部署很实用的方案，但如果要多实例部署，需要改成 Postgres、Supabase 或其他共享存储。

`lib/intelligence` 和 `lib/community-intel` 是更复杂的 agent/评估/闭环模块。它们试图把观察事件、线索、行动建议、社区帖子、RAG、外联角色等整合起来。现在更像内部实验能力，不应作为公开销售承诺的核心。

## 线索数据流

LeadPulse 的数据流可以拆成六步。

第一步是原始发现。数据可能来自小红书、知乎、微博、豆瓣、论坛、搜索结果、社群、OpenClaw、Gemini Deep Research 或人工收集。早期文档中多次提到 Gemini 的角色：用 Deep Research 搜索潜在客户，再由本地 importer 转成结构化线索。

第二步是结构化。原始内容会被整理成 platform、author、keyword、content、author_url、post_url、source_url、collected_at 等字段。这个结构对于后续去重、评分、筛选和导出都很重要。

第三步是意图评分。`app/api/leads/route.ts` 中有 vertical config，比如 `study_abroad` 和 `vibe_coding`。它用 intentKeywords、targetHints、competitorKeywords 判断意图强弱、是否目标客户、是否竞品/机构。它还看需求词、问题词、急迫词、是否有主页链接等因素，最终生成 score、intent_level、is_target、is_competitor、dm_ready。

第四步是过滤展示。前端默认会请求 `minScore=65`、`onlyTarget=1`、`excludeCompetitors=1`。用户也可以在后台线索页调整最低分、是否仅目标客户、是否排除竞品、返回数量。结果会显示平台、作者、分数、意图、关键词、可私信、证据摘要。未解锁时链接被隐藏。

第五步是导出解锁。用户触发导出后，系统检查 wallet 权限，成功则返回 CSV，包含 author_url、post_url、source_url 等可继续联系的字段，同时更新 wallet 的导出次数和链接解锁时间。当前默认链接解锁小时数来自 `LINK_UNLOCK_HOURS`，默认 72 小时。

第六步是触达跟进。线索拿到后，可以生成消息草稿、复制私信或邮件、进入任务队列、标记已发送、继续推进。这个部分决定产品是否只是“名单工具”，还是能成为“成交辅助系统”。

## 线索评分逻辑

LeadPulse 当前评分不是复杂机器学习模型，而是一套可解释的规则评分。这个阶段用规则是合理的，因为早期最重要的是可控、可调、可解释。

它会从内容中检查留学、申请、文书、中介、择校、雅思、托福、GPA、offer、硕士、本科、博士、费用、预算、签证、deadline 等关键词。如果命中多个词，或命中关键词同时出现急迫信号，会判定为 high intent。只有一个关键词或问题表达则是 medium。否则是 low。

它会识别需求词，例如求推荐、求助、请问、有没有、哪家、想找、怎么选、预算、费用、急、避雷。它也识别问题词，例如问号、请问、有没有、哪家、怎么、如何。这样的词往往比行业关键词更接近真实需求。

它会识别竞品或噪声。作者名如果包含中介、机构、顾问、工作室、教育、老师、官方、留学服务、留学咨询，容易被判定为机构号。内容如果包含私信我、加我微信、欢迎咨询、服务报价、套餐、保录、代办等，也会偏向竞品/营销内容。这个逻辑很关键，因为客户最反感的是买到一堆同行广告。

它会检查是否可私信。比如小红书用户主页 URL、access hint 中的 dm_ready 等。可私信会增加分数，因为可行动性更高。

这套逻辑的优点是快、透明、可改。缺点是容易误判，尤其是中文语境复杂、反讽、转述、经验分享和真实求助之间很难靠规则完全区分。因此最佳使用方式不是完全自动决策，而是“先粗筛，再人工确认”。这也符合代跑版的价值。

## 商业模式

当前定价设计很明确：免费样本、软件版、代跑版。

免费样本是获客入口。它不应该承诺太多，只需要让客户看到 5 条左右真实样本，确认方向是否匹配。免费样本的目标不是交付完整价值，而是让客户从“这东西是不是骗局”转成“这些人确实像我的潜在客户”。

软件版 ¥99/月适合愿意自己动手的人。客户可以查看和筛选名单、导出可继续联系的结果、获得基础私信参考。这个价格很低，适合早期收第一批付费用户，验证是否有人愿意为线索筛选付费。它的利润率来自自动化，但客服和数据成本不能忽略。

代跑版 ¥999/首轮适合没时间折腾的人。你帮客户跑一轮、筛掉明显噪声、交付名单和首轮沟通建议。这个产品形态更接近服务，早期更容易成交，因为客户买的是结果，不是工具学习成本。代跑版也能反哺软件：你在代跑过程中知道哪些筛选规则有效、哪些平台数据质量高、客户真正关心哪些字段。

未来可以扩展成几个层级。第一，免费样本继续保留。第二，轻量软件版按月收费。第三，行业代跑包按轮收费，例如留学、代运营、B2B SaaS、法律咨询、医美、财税。第四，高客单托管增长服务，按月或按结果收费。第五，如果数据和流程稳定，再做真正自助 SaaS。

## 为什么不应该一开始就做纯 SaaS

纯 SaaS 要求用户能自己完成从注册、理解产品、配置关键词、接入数据源、筛选线索、导出、触达、复盘的全流程。LeadPulse 当前的用户群体很多并没有这个耐心。服务工作室老板更关心“你帮我找到了谁”，而不是“你的过滤器有多少参数”。

另外，线索数据质量不是稳定商品。不同平台、关键词、时间、行业，结果差异很大。你必须不断人工看样本、调关键词、调排除规则。纯 SaaS 会把这些复杂性甩给用户，用户一旦拿到噪声数据就流失。

代跑服务可以把复杂性留在内部，把交付包装成结果。客户只看最终名单和建议。等你跑过足够多轮，知道哪些场景可重复，再把重复部分产品化。这个顺序更现实。

## 当前最大优势

第一，定位足够具体。它不是“AI 自动增长平台”这种抽象概念，而是“从公开平台找正在找方案的人”。这个表达容易理解，也容易验证。

第二，产品路径短。免费样本、看收费、先聊 15 分钟、开通软件版、预约代跑，都是可执行动作。没有把用户扔进复杂后台。

第三，数据结构已经跑通。线索 API 能返回 summary、rows、entitlements、source、locked_link_count。导出 CSV 已经有字段设计。后台能筛选、保存视图、自动刷新。

第四，早期商业形态合理。¥99 和 ¥999 的组合能同时覆盖自助和代跑。更重要的是，它允许你边卖边学习，而不是先闭门造一个完整平台。

第五，仓库里有很多可复用资产。虽然混乱，但有爬虫、导入器、邮件生成、触达、评分、Streamlit、Next、后端、部署脚本、文档。只要整理主线，就不是从零开始。

## 当前最大问题

第一，线上部署和 GitHub HEAD 不一致。线上站点标题仍是“公开平台销售线索平台”，本地当前 HEAD 是“帮你找到正在找服务的人”。线上 `/product` 和 `/pricing` 的标题也更像旧提交。这会导致你以为代码已经改了，但线上没有生效。必须建立明确部署流程。

第二，权限模型还很轻。页面有 internal access，但 API 需要逐个检查。此前 `/api/export` 可以匿名导出 CSV，已经在本地修复，但线上必须部署后才生效。未来所有敏感 API 都应该统一鉴权。

第四，后端不可健康构建。`backend` 当前 `npm run build` 会失败，原因包括 Prisma 7 不再支持 schema 中 datasource url、Stripe API 版本类型不匹配、LinkedIn 服务 DOM 类型缺失、隐式 any 等。这说明后端不是当前可依赖主路径。

第五，数据合规风险。公开平台线索采集涉及平台条款、隐私、反骚扰和商业使用边界。产品文案必须保持克制，不能承诺“批量抓取私域联系方式”或“自动骚扰”。交付时也要提醒客户遵守平台规则。

第六，产品叙事有多条线混杂。GuestSeek、LeadPulse、留学获客、B2B SaaS 获客、设计伙伴、增长前端、OpenClaw、自用闭环，这些都是历史阶段。对外必须只保留一条主线：公开平台意向线索和首轮触达。

## 生产环境必须配置的变量

`LEADPULSE_INTERNAL_ACCESS_KEY` 是内部页面访问密钥。没有它，生产后台要么不可访问，要么存在配置错误风险。这个值应使用随机长字符串。

`WALLET_SIGNING_SECRET` 是 wallet token/cookie 的签名密钥。它必须是随机长字符串，不能用默认值，不能提交到 GitHub。

`FREE_EXPORT_LIMIT` 控制新用户免费导出次数。生产建议默认 `0`，不要随意放开。如果要给试用客户导出，应通过兑换码、后台发积分或人工发样本，而不是开放匿名导出。

`DEFAULT_EXPORT_CREDITS` 控制新用户默认积分。生产建议默认 `0`。如果要做活动，可以临时配置，但要配合速率限制和用户标识。

`EXPORT_CREDIT_COST` 是单次导出消耗，默认 20。它可以以后和套餐对应。

`LINK_UNLOCK_HOURS` 是导出后链接解锁时间，默认 72 小时。这个设计可以防止永久公开链接。

`NEXT_PUBLIC_SITE_URL` 应配置成 `https://leadpulse.cc.cd`，否则付款、启动页、通知链接会生成错误 URL。

`LEADPULSE_PAYMENT_PROVIDER` 默认走 `wechat` 人工核验。如果要启用 Stripe，设置为 `stripe`，并配置 `STRIPE_SECRET_KEY` 和 `STRIPE_WEBHOOK_SECRET`。

`LEADPULSE_REDEEM_CODES` 用于启动码/兑换码。它应该只在生产环境变量中配置，不要写入代码。

通知相关变量包括 `LEADPULSE_SLACK_WEBHOOK_URL`、`LEADPULSE_FEISHU_WEBHOOK_URL`、`LEADPULSE_GOOGLE_APPS_SCRIPT_URL` 等。它们用于付款意向、预约、设计伙伴申请等通知。

## 部署关系

仓库中 `ops/public_deploy/README.md` 提到固定域名部署脚本：如果 `leadpulse.cc.cd` 在 Linux 服务器上通过 Nginx 反代到本机 3005 端口，可以用 `ops/public_deploy/deploy_fixed_domain.sh`。脚本会拉取 `origin/main`，在 `frontend-b2b` 执行 `npm ci`，重新 build，杀掉旧的 `next start --hostname 127.0.0.1 --port 3005`，再重启。

这说明线上很可能不是 Vercel，而是某台服务器上的 Next start，由 Nginx 或 Cloudflare 暴露。线上 API 返回的 `source_detail` 曾显示 `/opt/leadpulse/frontend-b2b/data/leads_snapshot.json`，也支持这个判断。

正确部署步骤应该是：先把本地提交推送 GitHub；登录服务器；进入服务器仓库目录；确认环境变量；执行固定域名部署脚本；检查 `/api/health`；检查 `/api/export` 匿名请求是否返回 402；检查 `/dashboard` 门禁；检查首页标题是否更新。

如果服务器仓库不是 `$HOME/LeadPulse`，要设置 `REPO_ROOT=/var/www/LeadPulse` 之类的变量运行脚本。部署后应该写入 runtime log，方便查当前进程。

## 推荐的产品主线

未来三个月不要把所有历史模块都往外讲。对外只讲一条线：公开平台意向线索，先看样本，再自己用软件或交给我们代跑。

官网首页应该继续保持短。标题建议是“帮你从公开平台找到正在找方案的人”。副标题可以是“把评论区、帖子和公开讨论里的求推荐、问价格、问怎么选，整理成可筛选、可导出的首轮线索”。按钮保留“免费拿样本”“看收费方式”“先聊 15 分钟”。

产品页不要讲 agent、RAG、workflow、OpenClaw、Gemini、DeepSeek、Stripe 等内部实现。客户不关心。客户关心：你从哪里找、怎么筛、给我什么、我怎么联系、如果不准怎么办。

定价页保留早期价，但要更明确交付边界。免费样本给 5 条左右；软件版支持查看、筛选、导出、基础话术；代跑版交付一版名单和首轮沟通建议，不承诺成交，不默认自动续费。

后台只给内部或付费用户看。不要把后台变成公开 demo。公开 demo 可以用静态样本和遮罩，不暴露真实链接。

## 代跑服务 SOP

第一步，明确客户行业。不要一上来就跑。先问客户服务对象、客单价、地区、关键词、负面排除词、是否能私信、是否有官网/微信/预约入口。

第二步，构造关键词。每个行业至少有三类词：需求词、痛点词、比较词。留学行业例如“留学中介推荐”“文书老师”“申请预算”“避雷机构”“怎么选中介”。代运营行业例如“小红书代运营多少钱”“账号没人看怎么办”“求推荐代运营”“投流没效果”。B2B SaaS 可以是“有没有工具”“替代方案”“自动化怎么做”。

第三步，抓取候选样本。可以用 OpenClaw、平台搜索、Gemini Deep Research、人工搜索等方式。早期不要追求全自动，先保证每一批样本有质量。

第四步，清洗和去重。去掉机构号、广告号、营销号、同行、重复帖子、内容过短、无上下文样本。

第五步，评分。按意图强度、问题明确度、急迫性、可联系性、匹配度、噪声风险打分。保留理由。

第六步，交付样本。免费样本只给少量，展示内容摘要和判断理由，链接可以隐藏或部分隐藏。付费导出才给完整链接。

第七步，生成触达建议。不要给客户群发模板。每条高分线索至少给一个首轮私信角度，例如先回应对方问题，再给一个小建议，再邀请进一步评估。

第八步，复盘。记录哪些关键词命中好、哪些平台噪声高、哪些话术有回复。下一轮调整关键词和评分规则。

## 自助软件版体验应该怎样设计

用户进入后台后，第一屏不要是复杂图表，而是线索总量、目标线索、高分线索、可私信主页四个指标，再给“筛选潜在客户”“查看消息草稿”“推进任务”几个动作。

线索页要让用户快速筛。保留最低分、返回数量、仅目标客户、排除竞品、快速预设、保存视图、自动刷新。这些当前已经有雏形。

导出按钮必须清楚说明成本和权限。比如“导出并解锁链接，本次消耗 20 积分”。如果没有权限，提示购买软件版或输入兑换码，不要直接报技术错误。

链接锁定是对的。未付费用户可以看到平台、作者、分数、意向、摘要，但看不到完整 author_url 和 post_url。这样既展示价值，又保护数据资产。

消息草稿页应该把草稿分成私信、邮件、跟进、成交推进。客户最容易用的是私信，不要默认让小团队配置 SMTP。

任务页应该把动作变成简单队列：今天联系谁、复制哪条话术、是否已发送、下一次什么时候跟进。不要把 CRM 做重。

## 数据合规与风控

LeadPulse 必须明确边界。它应只处理公开可见内容，不应承诺获取私人联系方式，不应绕过平台登录或反爬机制，不应鼓励客户批量骚扰，不应出售敏感个人信息。

导出内容应该尽量以公开帖子链接、作者主页、内容摘要、需求判断为主。对于手机号、微信、邮箱等字段，如果不是用户公开发布且合规可用，应谨慎处理。早期甚至可以不主动提取联系方式，只交付公开链接和触达建议。

触达建议应该强调个性化、低频、尊重平台规则。不要生成欺骗性话术，不要假装认识对方，不要制造虚假紧迫感。文案应回应对方公开问题，提供一点真实建议，再邀请对话。

后台需要保留导出记录。谁在什么时候导出了多少条，来源是什么，是否解锁链接。这不仅是商业计费，也用于风控。

如果要公开销售，隐私政策和服务条款要更严谨。当前 `lib/pricing.ts` 里的条款已经有基础，但需要法律层面再打磨，尤其是数据来源、客户使用责任、退款边界和禁止转售。

## 技术债清单

第一，后端构建失败。需要决定 `backend` 是继续维护还是归档。如果维护，要锁定 Prisma 版本到 6 或迁移到 Prisma 7 新配置；修复 Stripe API 版本；为 LinkedIn 服务补 DOM lib 或隔离浏览器代码；修复 TS strict 错误。

第二，主线代码分散。Next 前台、Streamlit、Express 后端、Python 脚本同时存在，容易让新接手的人迷失。建议在根目录 README 明确：当前线上主路径是 `frontend-b2b`，其他目录为 legacy/internal tools。

第三，数据源策略需要统一。当前 API 支持 local exporter、OpenClaw JSON、Supabase、snapshot，但生产实际走 snapshot。要明确线上到底用什么：如果是 snapshot，就建立定时生成和部署流程；如果是 Supabase，就接入服务端 key；如果是 OpenClaw，就保证服务器上有数据文件和定时任务。

第四，权限系统需要统一。现在 internal access、wallet token、redeem code、credits、dashboard 都是不同机制。短期可以接受，但敏感 API 必须全部检查权限。中期应该有统一 session/user/subscription。

第五，支付闭环半自动。当前微信收款 + 启动码适合早期，但要减少人工漏单。至少要有付款意向记录、通知、核验任务、发码记录、开通记录。Stripe 可以以后再接，不要先为国际支付复杂性拖慢销售。

第六，测试覆盖不足。现在主要靠构建和手动 API 验证。应该给 `/api/leads`、`/api/export`、`lead_wallet`、`redeem-code`、`payment-intent` 加最小单元/集成测试。

## 如何对客户介绍

你可以这样讲：

“LeadPulse 帮你从公开平台里找正在找方案的人。比如有人已经在评论区或帖子里问‘有没有靠谱推荐’‘多少钱’‘怎么选’‘怎么避坑’，这类人比普通流量更接近成交。我们先帮你整理一小批样本，你看方向对不对。如果对，你可以用软件自己筛和导出，也可以交给我们代跑一轮。”

“我们只处理公开可见内容，重点不是无限抓数据，而是把公开讨论里的真实需求整理出来，并尽量筛掉同行、广告和噪声。”

客户问“能不能保证成交”时，要克制：

“不能保证成交。我们交付的是更接近成交的线索和首轮沟通建议。成交还取决于你的服务、报价、响应速度和后续跟进。”

客户问“为什么先看样本”时：

“因为每个行业的数据质量不一样。先看样本最公平，如果样本不对，你不用继续买；如果样本对，再决定自己用软件还是让我们代跑。”

客户问“和 CRM 有什么区别”时：

“CRM 管你已经有的客户，LeadPulse 帮你先找到可能值得联系的人。它更靠前，解决的是线索来源和首轮触达。”

## 如何对自己判断产品是否值得继续

不要用访问量判断。早期最重要的指标是样本到付款的转化。

第一，看客户看到样本后是否愿意继续聊。如果 10 个目标客户里没有 1 个愿意聊，说明样本方向或客户选择有问题。

第二，看客户是否愿意为代跑付费。如果很多人说“有意思”但没人付 ¥999，说明痛点不够急、客群不对、交付不可信或价格/承诺不清。

第三，看每轮代跑是否可复用。如果每个客户都要完全人工从零研究，软件化价值低；如果关键词、过滤、评分、交付模板能复用，产品价值高。

第四，看线索是否真的能产生回复。即使没有成交，只要能产生有效回复，就说明方向有价值。

第五，看客户是否愿意复购。获客工具最大的价值是持续性。如果客户只买一次尝鲜，不续做，要查原因：数据不准、联系不到、话术无效、后续跟进麻烦，还是客户自己的服务不行。

## 仓库治理建议

根目录新增一个明确的 `CURRENT_PRODUCT.md` 或把本说明作为主入口。README 第一屏写清楚：当前线上主应用是 `frontend-b2b`，部署站点是 `leadpulse.cc.cd`，其他目录是 legacy 或 internal。

旧文档不要全删，但要归档。可以建 `docs/archive`，把 GuestSeek、早期 Streamlit、旧部署说明移进去。保留 `docs/ops`、`docs/product`、`docs/security`。

密钥治理要严格。任何文档中都不要写真实 key。所有 `.env` 只保留 `.env.example`。如果历史提交里有真 key，必须在对应服务商后台轮换。

依赖要锁。前端有 `package-lock.json`，部署用 `npm ci`。后端目前 Prisma 版本问题明显，要么修，要么先标注 legacy，不要让部署脚本依赖后端成功。

每次部署后写一条 release note：提交 hash、部署时间、环境变量变更、验证结果。这样以后不会再出现“我也记不得了”的状态。

## 这份产品的真实评价

LeadPulse 的方向是有商业感的。它抓住了小团队获客里最痛的一段：不是管理客户，而是找到值得联系的人。它也没有一开始就做重型系统，而是用免费样本、低价软件版、代跑版切入，这符合早期产品规律。

但它现在最大的问题不是缺功能，而是需要收束。仓库里有太多历史尝试，如果每条线都继续做，会变成一个谁也讲不清的系统。应该明确：当前主线是 `frontend-b2b` + 公开平台线索 + 样本/导出/代跑。Streamlit 和 Express 后端先作为历史资产，不要拖累主产品。

如果你要把它真正卖出去，下一步不是继续加 AI，而是把样本质量、导出权限、付款开通、代跑交付和部署一致性打磨到可靠。一个客户愿意为 20 条高质量线索付 ¥999，比一个复杂 dashboard 更能证明产品价值。
